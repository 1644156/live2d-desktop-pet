# Live2D 最终口型同步方案

## 1. 方案目标

本方案面向当前项目中的 Python + PyQt5 + QtWebEngine + pixi-live2d-display 架构，目标是在 **不依赖外部口型工具**、**不要求 TTS 返回音素时间戳** 的前提下，尽可能实现自然、稳定、可扩展的 Live2D 口型同步。

方案的最终目标不是追求严格的音素级强制对齐，而是实现以下三点：

1. **嘴巴跟随真实音频变化**，而不是按固定节奏机械开合。
2. **充分利用模型已有能力**，自动适配不同模型的参数配置。
3. **在大多数只有 `ParamMouthOpenY` / `ParamMouthForm` 的模型上也能显著提升口型自然度**，并在支持 `ParamA/I/U/E/O` 的模型上进一步升级到多元音驱动。

---

## 2. 当前项目现状

### 2.1 原始实现的问题

项目最初的口型同步实现并不基于真实音频，而是通过时长驱动一个假波形：

- Python 侧在 [src/app/chat_manager.py](src/app/chat_manager.py) 中拿到一句 TTS 音频的总时长后调用桥接接口。
- `WebBridge.start_lip_sync()` 在 [src/app/web_bridge.py:81](src/app/web_bridge.py:81) 到 [src/app/web_bridge.py:89](src/app/web_bridge.py:89) 把时长传给前端。
- 前端 `startLipSync()` 在 [public/desktop.html:440](public/desktop.html:440) 附近用 `Math.sin()` 人为构造一个周期性张嘴曲线。

这种方式的核心缺陷：

1. **嘴巴运动与真实发音无关**。
2. **停顿时仍可能继续动嘴**。
3. **无法区分不同发音形状**。
4. **所有模型的表现几乎一样，缺乏个体差异**。

### 2.2 当前 TTS 和音频链路的优势

虽然 TTS 不返回时间戳，但项目本身已经具备一个很重要的条件：

- TTS 是流式生成的。
- 播放器拿到的是实时 PCM 数据。
- PCM 在进入声卡前可以被同步分析。

当前关键链路：

- TTS 服务接收音频增量：`response.audio.delta`
- 播放器在 [src/tts/audio_player.py](src/tts/audio_player.py) 中以 PCM 方式播放
- 采样率固定为 `24000 Hz`，16-bit，单声道

这意味着：

> 即使没有音素时间戳，我们仍可以对真实播放中的音频帧做实时分析，并把分析结果直接驱动 Live2D 嘴部参数。

## 3. 最终方案总览

### 3.1 核心思路

最终方案采用 **实时 PCM 音频分析 + 模型能力自动检测 + 多级降级适配**。

整体链路：

```text
TTS 生成 PCM 流
-> AudioPlayer 播放 PCM
-> 同时把 PCM chunk 喂给 LipSyncAnalyzer
-> 提取 RMS / ZCR / 频带能量
-> 计算 open_y / mouth_form / a i u e o
-> 通过 Qt 信号发回主线程
-> WebBridge 调用 JS updateMouth(...)
-> JS 根据模型能力选择驱动方式
-> Live2D 模型更新嘴部参数
```

### 3.2 为什么这是当前最可行的方案

和“上外部强制对齐器”相比，本方案的优势：

1. **不依赖外部工具**，部署简单。
2. **直接复用当前项目已有 TTS 和播放链路**。
3. **对大多数标准模型立即有效**。
4. **对 Mao 这类高级模型可自动升级**。
5. **可以在现有结构上平滑落地，不需要推翻重做。**

## 4. 模型能力分级与兼容策略

这是方案里最关键的设计之一。

### 4.1 为什么不能只看 `.model3.json`

`.model3.json` 不是模型参数的完整来源，它主要负责资源清单、分组和配置。
真正的参数定义在 `.moc3` 中。

因此会出现一种常见情况：

- `.model3.json` 里没写 `ParamI/U/E/O`
- 但 `.moc3` 里实际存在这些参数
- Cubism Viewer 能看到
- 运行时 `coreModel.getParameterIndex()` 也能拿到

Mao 模型就是这种情况：

- `Mao.model3.json` 的 LipSync 分组只绑定了 `ParamA`
- 但 `Mao.moc3` 实际存在 `ParamA/I/U/E/O`

所以最终方案不依赖 `.model3.json` 静态判断，而是在运行时直接检测 `coreModel`。

### 4.2 运行时能力分级

前端在模型加载完成后调用 `detectMouthCapabilities()`，按优先级检测以下能力：

#### 模式 1：`full_viseme`
适用条件：

- 检测到 3 个及以上元音参数
- 候选参数包括：
  - `ParamMouthA` / `ParamA`
  - `ParamMouthI` / `ParamI`
  - `ParamMouthU` / `ParamU`
  - `ParamMouthE` / `ParamE`
  - `ParamMouthO` / `ParamO`

行为：

- 同时驱动 `a/i/u/e/o` 权重
- `openY` 仍然用于整体张开度
- 适合 Mao 这类高级模型

#### 模式 2：`open_form`
适用条件：

- 检测到 `ParamMouthOpenY`
- 检测到 `ParamMouthForm`

行为：

- `openY` 控制开合
- `mouthForm` 控制嘴形偏扁或偏圆
- 这是大多数模型的主力模式

#### 模式 3：`open_only`
适用条件：

- 仅检测到 `ParamMouthOpenY`（或别名）

行为：

- 只控制张开与闭合
- 不驱动嘴形变化

#### 模式 4：`unsupported`
适用条件：

- 检测不到任何可用嘴部参数

行为：

- 禁用口型同步
- 不报错，不影响模型其他功能

### 4.3 运行时检测优于静态配置

这种做法有几个直接好处：

1. **自动适配不同模型**，不用手工给每个模型写特殊逻辑。
2. **能发现 `.moc3` 中存在但 `.model3.json` 未声明的参数**。
3. **降低模型资源不规范带来的问题**。
4. **未来换模型时不需要大改代码。**

## 5. 音频分析算法设计

### 5.1 输入数据

`LipSyncAnalyzer` 接收来自播放器的 PCM 数据，格式固定为：

- 采样率：24000 Hz
- 声道：单声道
- 位深：16-bit signed integer

每个 PCM chunk 在 [src/tts/audio_player.py](src/tts/audio_player.py) 中播放前同步喂给分析器。

### 5.2 提取的特征

为了不依赖 `numpy` / `scipy` / `librosa`，方案只使用 Python 标准库和 `array` 模块提取三类特征。

#### 5.2.1 RMS
实现位置：[src/tts/lipsync_analyzer.py:7](src/tts/lipsync_analyzer.py:7)

作用：

- 描述当前音频块的响度
- 直接用于控制 `open_y`

结论：

- 声音越大，嘴巴张得越大
- 静音或接近静音时闭嘴

#### 5.2.2 ZCR（过零率）
实现位置：[src/tts/lipsync_analyzer.py:17](src/tts/lipsync_analyzer.py:17)

作用：

- 粗略区分清辅音/摩擦音与浊元音
- 高 ZCR 倾向于扁嘴音（例如 s / x / sh 类）

#### 5.2.3 频带能量分布
实现位置：

- Goertzel 能量计算：[src/tts/lipsync_analyzer.py:29](src/tts/lipsync_analyzer.py:29)
- 频带比值计算：[src/tts/lipsync_analyzer.py:46](src/tts/lipsync_analyzer.py:46)

当前取样频点：

- 低频：500 Hz
- 中频：1200 Hz
- 高频：2500 Hz

作用：

- 粗略判断当前更接近 A / I / U / E / O 哪一类发音区间
- 配合 `mouthForm` 和元音权重驱动不同嘴形

### 5.3 口型参数推导逻辑

核心计算在 [_compute_viseme()](src/tts/lipsync_analyzer.py:59)。

输出为 7 个值：

```text
(open_y, mouth_form, a, i, u, e, o)
```

逻辑摘要：

- `rms < 0.01`：视为静音，所有参数归零
- `zcr > 0.22`：偏摩擦音，`mouth_form` 偏正，`i/e` 权重提高
- `low > 0.6`：偏圆唇音，`mouth_form` 偏负，`o/u` 权重提高
- `mid > 0.45`：偏 A 类张口音，`a` 权重提高
- `high > 0.4`：偏 I/E 类前元音，`mouth_form` 偏正，`i/e` 提高
- 其他情况：默认轻度 A 类口型

这是一个**规则驱动的近似音素映射**，不是严格语言学建模，但对于 TTS 场景已经足以显著提升自然度。

## 6. 平滑策略

如果直接把每一帧分析结果原样赋给模型，嘴巴会明显抖动。为此方案加入了专门的平滑层 `_LipSyncSmoother`。

实现位置：[src/tts/lipsync_analyzer.py:103](src/tts/lipsync_analyzer.py:103)

### 6.1 快开慢闭

对 `open_y` 使用不同速度：

- `attack = 0.6`：目标比当前大时，加快追踪
- `release = 0.15`：目标比当前小时，慢速回落

效果：

- 张嘴更灵敏
- 收嘴更自然
- 减少抽搐感

### 6.2 统一平滑嘴形与元音权重

- `mouth_form` 使用 `form_speed = 0.25`
- `a/i/u/e/o` 也用同样节奏做指数平滑

效果：

- 嘴形变化不会突变
- 元音切换更柔和

### 6.3 更新频率限制

`LipSyncAnalyzer` 内部通过 `MIN_INTERVAL = 0.025` 把更新频率限制在大约 40 FPS。

优点：

- 降低 CPU 开销
- 避免过高频率导致 GUI 线程消息过密
- 对视觉效果已经足够流畅

## 7. 前后端落地实现

### 7.1 Python 侧：LipSyncAnalyzer

文件：[src/tts/lipsync_analyzer.py](src/tts/lipsync_analyzer.py)

职责：

1. 接收 PCM bytes
2. 解析成 `array("h")`
3. 提取 RMS / ZCR / 频带能量
4. 计算 7 维口型目标值
5. 做平滑
6. 通过 `visemeUpdated` 信号发出结果

信号签名：

```python
pyqtSignal(float, float, float, float, float, float, float)
```

即：

```python
(open_y, mouth_form, a, i, u, e, o)
```

### 7.2 Python 侧：AudioPlayer

文件：[src/tts/audio_player.py](src/tts/audio_player.py)

新增能力：

- `set_lipsync_analyzer()` 注入分析器
- 在 `_play_loop()` 中播放 PCM 前调用 `self._lipsync_analyzer.feed(audio_data)`

这样做的关键点：

- 分析与播放使用同一份 PCM 数据
- 不需要额外复制或重新解码音频
- 口型更新与真实播放保持高度同步

### 7.3 Python 侧：ChatManager

文件：[src/app/chat_manager.py](src/app/chat_manager.py)

改造点：

1. 初始化时创建 `LipSyncAnalyzer`
2. 将 `visemeUpdated` 连接到 `_on_viseme_updated()`
3. TTS 初始化成功后，将分析器注入 AudioPlayer
4. 音频准备好时调用：
   - `self._window._bridge.start_real_lip_sync()`
   - `self._lipsync_analyzer.start()`
5. 音频完成或流被中断时调用：
   - `self._lipsync_analyzer.stop()`
   - `self._window._bridge.stop_real_lip_sync()`

这样就替代了旧的 `start_lip_sync(duration)` 主流程。

### 7.4 Python 侧：WebBridge

文件：[src/app/web_bridge.py](src/app/web_bridge.py)

新增接口：

- `start_real_lip_sync()`
- `stop_real_lip_sync()`
- `update_mouth(open_y, mouth_form, a, i, u, e, o)`

其作用是将 Python 分析结果转成前端可调用的 JS 函数。

### 7.5 前端：desktop.html

文件：[public/desktop.html](public/desktop.html)

关键新增函数：

- `detectMouthCapabilities()`
- `startRealLipSync()`
- `stopRealLipSync()`
- `updateMouth(openY, mouthForm, vA, vI, vU, vE, vO)`

关键机制：

#### 7.5.1 使用 `mouthSync` 驱动开合

为了和 `pixi-live2d-display` 的内部逻辑兼容，本方案不直接每帧硬写 `ParamMouthOpenY`，而是：

- 在进入实时口型模式时，替换 `motionManager.mouthSync`
- 让它返回我们实时更新的 `currentMouthOpenY`

这样可以充分复用库内置的 LipSync 更新机制。

#### 7.5.2 直接写 `mouthForm` 和元音参数

`mouthForm` 与 `a/i/u/e/o` 不受 `mouthSync` 机制管理，因此在 `updateMouth()` 中直接调用：

```javascript
core.setParameterValueById(...)
```

#### 7.5.3 停止时统一归零

`stopRealLipSync()` 会重置：

- `mouthSync`
- `mouthForm`
- 所有可用的元音参数

避免嘴巴停留在奇怪形态。

## 8. 针对不同模型的最终效果

### 8.1 只有 `ParamMouthOpenY`

效果：

- 嘴巴跟真实声音开合
- 停顿时闭嘴
- 声音大小不同，张嘴幅度不同

相比旧方案已经明显提升。

### 8.2 有 `ParamMouthOpenY + ParamMouthForm`

效果：

- 除了开合，还能区分“扁嘴”和“圆嘴”
- 发 O/U 类音时嘴更圆
- 发 I/E/S 类音时嘴更扁

这正是大部分标准模型的最佳实现路径。

### 8.3 有 `ParamA/I/U/E/O`

效果：

- 进入 `full_viseme` 模式
- 可直接驱动多元音权重
- Mao 这类模型会出现更细腻的元音切换

需要强调：

> 即使 `.model3.json` 未声明这些参数，只要 `.moc3` 中存在，运行时检测仍然能发现并使用它们。

## 9. 为什么 `ParamMouthForm` 很重要

你观察到大部分模型都带有：

- `ParamMouthOpenY`
- `ParamMouthForm`

这是本方案成立的重要基础。

如果只控制 `ParamMouthOpenY`，嘴巴只是上下开合，看起来像“会动”，但不像“会发音”。

而 `ParamMouthForm` 可以承担一个很关键的职责：

- `> 0`：偏横向、偏扁嘴
- `< 0`：偏纵向、偏圆嘴

于是即便模型没有独立的 `ParamO`、`ParamU`，也可以通过 `mouthForm` 把“呜/哦”的圆嘴感模拟出来；同理，`I/E` 类音也能做出更扁的嘴型。

因此，对于绝大多数模型：

> `ParamMouthOpenY + ParamMouthForm` 就是本项目最关键、最值得利用的一组参数。

   - 说话时嘴巴是否张开
   - 停顿时是否闭嘴

1. 很短的音频片段

- 不残留旧口型
- 不报错

## 11. 调参建议

目前的默认参数适合作为第一版上线值：

- `attack = 0.6`
- `release = 0.15`
- `form_speed = 0.25`
- `MIN_INTERVAL = 0.025`
- RMS 静音阈值：`0.01`
- ZCR 扁嘴阈值：`0.22`

如果后续要微调，可按以下方向进行：

### 嘴巴不够灵敏

- 略微降低 RMS 满幅阈值
- 或增大 `attack`

### 嘴巴太抖

- 增大 `MIN_INTERVAL`
- 降低 `attack`
- 降低 `form_speed`

### 收嘴太慢

- 提高 `release`

### 圆嘴/扁嘴不明显

- 提高 `mouth_form` 的正负目标值
- 调整低频/高频阈值

## 12. 方案边界与后续升级

### 12.1 当前方案能做到什么

1. 比原先的假波形口型自然很多
2. 能跟着真实音频开合
3. 能利用 `ParamMouthForm` 近似模拟不同口形
4. 能在部分高级模型上启用 A/I/U/E/O 多元音驱动
5. 能自动适配不同模型，无需手工改每个模型文件

### 12.2 当前方案做不到什么

1. 不是严格音素级强制对齐
2. 不能保证每个汉字都映射到完全准确的嘴型
3. 对极复杂的连读、快速语流只能做到近似

### 12.3 后续可升级方向

如果未来需要继续提高精度，可考虑：

1. **引入外部音素对齐器**
   - Rhubarb Lip Sync
   - Whisper + 强制对齐器
   - 其他音频到音素时间轴方案

2. **切换到支持 viseme 时间戳的 TTS**
   - 直接拿到 TTS 返回的口型事件

3. **给模型补标准五元音参数**
   - 在 Cubism Editor 中补 `ParamMouthA/I/U/E/O`
   - 对大部分模型会有明显提升

---

## 13. 最终结论

结合当前项目结构、TTS 能力和模型资源情况，最终最合理、最可落地的口型同步方案是：

> **以实时 PCM 音频分析为基础，自动检测模型的嘴部参数能力，优先驱动 `ParamMouthOpenY + ParamMouthForm`，并在存在 `ParamA/I/U/E/O` 时自动升级到多元音模式。**

这套方案兼顾了：

- 可落地性
- 稳定性
- 模型兼容性
- 后续可扩展性

它不是理论上最强的方案，但在你当前这个项目里，它是**投入产出比最高、最适合直接上线和持续迭代的方案**。
